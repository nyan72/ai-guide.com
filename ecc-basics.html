Core Features

単体機能の解説 (Skills & Commands)

プロンプトを「資産化」し、特定のワークフローを短縮コマンドとして実行する仕組み。

1. Skills と Commands の違い

両者は似ていますが、保存場所と実行のされ方が異なります。簿記のように役割を整理すると以下の通りです。

名称 Skills (スキル) Commands (コマンド)
役割 広範なワークフローの定義
(ルール・制約に近い)		 即時実行されるプロンプト
(マクロに近い)
保存場所 ~/.claude/skills/ ~/.claude/commands/
実行方法 プロンプト内で自然に呼び出される /tdd のようにスラッシュで実行

2. 実践的な設定例

例：コードマップの更新 (Codemap Updater)

探索によるコンテキスト消費を抑えるため、チェックポイントごとにコードマップを更新するスキルです。

# ~/.claude/skills/codemap-updater.md
このスキルは、コードベースの構造を要約した
`codemap.md` を最新状態に保つためのものです。
ファイル構成に変更があった際に、自動的にツリー構造を更新します。

例：TDDワークフローのコマンド化

テストを先に書き、その後に実装を行う一連の流れを /tdd で呼び出せるようにします。

# ~/.claude/commands/tdd.md
1. 指定された機能のテストファイルを `tests/` に作成せよ
2. テストを実行し、失敗を確認せよ
3. テストをパスする最小限のコードを実装せよ
💡 プロのTips: コマンドの連結

これらのスキルとコマンドは、単一のプロンプト内で「連結」して実行可能です。
例：/refactor-clean を実行した直後に /test-coverage を回すといった自動化が可能です。

3. ファイル構造の可視化

これらの機能は、プロジェクト単位やユーザー単位で以下のように管理されます。

~/.claude/
├── skills/
│   ├── coding-standards.md   (コーディング規約)
│   └── security-review.md    (セキュリティチェックリスト)
└── commands/
    ├── tdd-workflow/         (フォルダによる構造化も可能)
    └── pmx-guidelines.md     (プロジェクト特有のパターン)

まとめ：なぜこれが必要か？

  • コンテキストの節約: 長い指示を毎回打つ必要がなくなり、トークンを節約できる。
  • 一貫性の確保: チームや個人で共通の「解き方」を固定できる。
  • 自動化の基盤: 後述する「Hooks」と組み合わせることで、特定の動作をトリガーにこれらのスキルを自動発動できる。
